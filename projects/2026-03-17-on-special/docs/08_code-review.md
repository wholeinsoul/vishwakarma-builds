## CRITICAL

### 1. Missing `increment_rate_limit` RPC function in schema
- **File:** `supabase/schema.sql` (missing) / `app/api/generate/route.ts:149`
- **Issue:** The generate endpoint calls `supabase.rpc("increment_rate_limit", ...)` but this function is never defined in the schema. Rate limiting will silently fail — the fallback upsert works but isn't atomic, creating a race condition where concurrent requests can exceed the daily limit.
- **Fix:** Added `increment_rate_limit` function to schema.sql. ✅ FIXED

### 2. Overly permissive RLS policy on subscriptions
- **File:** `supabase/schema.sql:88-91`
- **Issue:** The "Service role manages subscriptions" policy uses `USING (true) WITH CHECK (true)` with no role restriction. This applies to ALL roles including `anon` and `authenticated`, meaning any authenticated user can INSERT/UPDATE/DELETE any subscription row. The service role already bypasses RLS, so this policy is both redundant and dangerous.
- **Fix:** Dropped the dangerous policy. Service role bypasses RLS by default. ✅ FIXED

### 3. `social_handles` accepts arbitrary JSON — stored XSS risk
- **File:** `app/api/bar-profile/route.ts:42-44`
- **Issue:** `social_handles` is validated only as `typeof object && !== null`, then stored as JSONB and rendered in prompts and potentially in the UI. An attacker could store `{"instagram": "<script>alert(1)</script>"}` or inject prompt manipulation strings.
- **Fix:** Added sanitization to only allow string values and strip HTML. ✅ FIXED

## WARNING

### 4. Landing page is still default Next.js boilerplate
- **File:** `app/page.tsx:1-50`
- **Issue:** The root page shows the default "Create Next App" template with Vercel deploy links. This is dead code that should be replaced with an actual landing/marketing page for On Special.
- **Fix:** Replace with a proper landing page or redirect to `/dashboard`.

### 5. Root layout metadata is default boilerplate
- **File:** `app/layout.tsx:16-19`
- **Issue:** Title is "Create Next App" and description is "Generated by create next app". Should be "On Special" branding.
- **Fix:** Update metadata to match product. ✅ FIXED

### 6. Auth callback has open redirect vulnerability
- **File:** `app/auth/callback/route.ts:5`
- **Issue:** The `next` query parameter is used directly in a redirect without validation. An attacker could craft a URL like `/auth/callback?code=...&next=https://evil.com` to redirect users after login.
- **Fix:** Validate that `next` starts with `/` and doesn't contain `//`. ✅ FIXED

### 7. No request body size limit on generate endpoint
- **File:** `app/api/generate/route.ts:37`
- **Issue:** `request.json()` has no try/catch — a malformed body will throw an unhandled exception (caught by outer catch, but returns a misleading 503 instead of 400).
- **Fix:** Wrap `request.json()` in try/catch with proper 400 response. ✅ FIXED

### 8. `document.execCommand("copy")` is deprecated
- **File:** `components/copy-button.tsx:15-19`
- **Issue:** The fallback copy mechanism uses `document.execCommand("copy")` which is deprecated and removed in some browsers. Not harmful, but the fallback may silently fail.
- **Fix:** Consider using a polyfill or showing a "manual copy" tooltip as fallback.

### 9. History page `useEffect` missing proper dependency
- **File:** `app/(dashboard)/history/page.tsx:18-20`
- **Issue:** `fetchHistory` is called in `useEffect` with `[page]` dependency, but ESLint would flag `fetchHistory` as a missing dependency. This is functionally correct but not best practice.
- **Fix:** Move fetch logic inline or use `useCallback`.

### 10. Stripe webhook doesn't verify event idempotency
- **File:** `app/api/stripe/webhook/route.ts`
- **Issue:** Duplicate webhook deliveries (which Stripe does retry) will process the same event multiple times. For most operations this is idempotent (upserts), but `invoice.paid` does an update that could race with `checkout.session.completed`.
- **Fix:** Store processed event IDs and skip duplicates, or accept the current behavior since upserts make it mostly safe.

### 11. Rate limit date uses server timezone, not UTC
- **File:** `app/api/generate/route.ts:78`
- **Issue:** `new Date().toISOString().split("T")[0]` uses UTC, but the SQL schema uses `CURRENT_DATE` which uses the database timezone. If the server and DB are in different timezones, the rate limit check and increment could target different dates.
- **Fix:** Ensure both use UTC consistently. Current code is fine if Supabase is UTC (default), but document this assumption.
